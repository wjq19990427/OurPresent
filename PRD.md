## 1. 产品概述

### 1.1 产品定义

**OurPresent** 是一款专为情侣设计的私密双人情感记录与智能辅助工具。其核心价值主张是：

> 在保护双方绝对个人隐私的前提下，通过"延时共享"机制让感情沉淀，并在合适的时机以温柔的方式让彼此看见对方的心意。

产品分两个阶段构建：
- **第一阶段**：双人私密记录空间，配备时间锁权限引擎与完整的关系生命周期管理。
- **第二阶段**：接入 LLM + RAG，引入 AI 智能体作为关系的客观记录者和情感疏导者。

### 1.2 产品定位

| 维度 | 定位 |
|------|------|
| 核心功能 | 情侣双人私密内容记录、延时共享、情感 AI 辅助 |
| 差异化 | 时间锁机制（90 天延时）+ 严格隐私隔离 + NVC 中立 AI |
| 技术路线 | 本地化部署（隐私优先）→ 后期可选云端迁移 |
| 当前形态 | 本地内网 Streamlit Demo |

### 1.3 产品命名含义

"Present" 在产品中承载三层递进含义：

1. **我们的当下（Our Present）**：记录此刻的真实状态，无论喜悦还是难以直言的委屈。
2. **共同的馈赠（The Gift of Today）**：两人相遇本是礼物，提醒珍惜当下的陪伴。
3. **温柔地表达（Present to You）**：每一条记录经时间沉淀后，化作一份延迟解锁的情感礼物。

## 2. 目标用户与使用场景

### 2.1 核心用户画像

**用户 A：记录型一方**
- 感情细腻，有记录日常的习惯
- 有些心里话当面难以开口，倾向于用文字整理情绪
- 在意隐私，不希望自己的情绪被第三方（闺蜜/兄弟）外传
- 希望对方有一天能看到自己的感受，但需要时间缓冲

**用户 B：被动接收型一方**
- 可能不擅长表达，但愿意了解对方的内心
- 希望在双方关系稳定、情绪冷静后再了解对方的想法
- 对隐私有一定要求，不希望被强制分享

### 2.2 典型使用场景

| 场景 | 描述 |
|------|------|
| 日常记录 | 上传两人的照片、视频，配上感受描述，作为共同回忆存档 |
| 情绪倾诉 | 发生争吵后，一方记录下自己的委屈和想法，暂时私密存放 |
| 延时告白 | 记录一段深情的文字，申请 90 天后让对方看到 |
| 关系复盘 | AI 智能体基于双方共享内容，生成客观的情感周报 |
| 情感疏导 | 向实时助手倾诉情感困惑，获得 NVC 框架下的中立回应 |
| 分手处理 | 关系结束时，平静有序地完成数据清理，保护双方隐私 |

## 3. 核心问题与解决思路

### 3.1 现有痛点

1. **隐私外泄风险**：情侣间的矛盾一旦倾诉给闺蜜/兄弟，存在信息外传和被放大的风险，甚至激化矛盾。
2. **表达时机错位**：情绪激动时说出的话往往伤人，但等情绪平静后又难以主动开口。
3. **缺乏客观视角**：双方各执一词，缺少一个中立、不偏袒任何一方的视角来梳理关系状态。
4. **记忆碎片化**：两人的日常照片、视频散落在各自手机，缺乏统一的、有情感温度的归档空间。

### 3.2 解决方案映射

| 痛点 | 解决方案 |
|------|----------|
| 隐私外泄 | 本地化部署 + 严格的双用户隔离，记录默认私密，不经授权对方不可见 |
| 表达时机错位 | 时间锁机制：申请共享后满 90 天才对对方可见，给情绪足够的缓冲期 |
| 缺乏客观视角 | AI 情感周报 Agent：只基于双方共享内容，用抽象指标客观呈现关系状态 |
| 记忆碎片化 | 双人记录空间：图片/视频/文字统一归档，强制填写情感描述，形成结构化记忆 |

## 4. 产品架构与阶段规划

### 4.1 技术架构

```
┌─────────────────────────────────────────┐
│              前端（Streamlit）            │  ← frontend/
├─────────────────────────────────────────┤
│              鉴权层（Auth）               │  ← backend/auth_manager.py
├─────────────────────────────────────────┤
│              数据层（DB）                 │  ← backend/db_manager.py
├───────────────────┬─────────────────────┤
│   结构化存储       │    向量存储（预留）    │
│  data/db.json     │   ChromaDB / Pinecone │
└───────────────────┴─────────────────────┘
```

**设计原则**：`backend/db_manager.py` 不感知 Streamlit，`backend/auth_manager.py` 不直接操作 `session_state`，三层完全解耦，后期前后端分离时后端模块可直接复用。

### 4.2 阶段规划

| 阶段 | 目标 | 关键交付 | 状态 |
|------|------|----------|------|
| **Alpha** | 本地可运行 Demo | 注册/登录、情侣绑定、记录上传、时间锁、分手协议 | ✅ 已完成 |
| **Beta** | 稳定内网版本 | 前后端分离、更强密码方案、并发安全、移动端适配 | 🔲 规划中 |
| **Phase 2** | AI 智能体接入 | 情感周报 Agent、实时情感助手（NVC）、RAG 向量检索 | 🔲 规划中 |
| **v1.0** | 正式发布 | 云端部署选项、完整测试覆盖、用户引导流程 | 🔲 规划中 |

## 5. 功能需求（第一阶段）

### 5.1 用户账户系统

#### FR-01 用户注册
- **描述**：新用户通过用户名 + 密码在网页端完成注册
- **规则**：
  - 用户名：2-20 字符，全局唯一
  - 密码：至少 6 位
  - 注册成功后系统生成唯一 `user_id`（格式：`usr_` + 8位 hex）
  - 注册成功页面明确展示 `user_id`，并提示用户将其分享给伴侣用于绑定

#### FR-02 用户登录
- **描述**：用户通过用户名 + 密码登录
- **规则**：
  - 登录成功后生成 24 小时有效期的 auth token，写入 URL query param
  - 刷新页面时自动通过 token 恢复登录状态，无需重新输入
  - token 到期后跳转至登录页
  - 主动点击「退出登录」立即撤销 token

#### FR-03 持久化登录
- **描述**：通过 URL query param 中的 token 实现跨页面刷新的登录保持
- **规则**：每个浏览器 tab 拥有独立 token，互不干扰

---

### 5.2 情侣绑定系统

#### FR-04 发送绑定请求
- **描述**：已登录用户在「账户」页输入对方 `user_id`，向其发送绑定邀请
- **规则**：
  - 目标用户不存在时提示错误
  - 双方均不能已有活跃/冻结的绑定关系
  - 不能向自己发送请求
  - 请求发送后，发起方看到「等待对方确认」提示

#### FR-05 接受/拒绝绑定
- **描述**：被邀请方登录后，在「账户」页的「收到的绑定请求」区看到邀请，可接受或拒绝
- **规则**：
  - 接受后双方 `couple_id` 更新，关系状态变为 `active`
  - 拒绝后请求记录删除，双方恢复无绑定状态

#### FR-06 关系状态流转
- **状态机**：`pending_bind` → `active` → `frozen` → `dissolved`

---

### 5.3 内容记录系统

#### FR-07 上传文件
- **描述**：用户可上传图片（jpg/png）、视频（mp4）、文本文件（txt/md）
- **规则**：
  - 同一批上传的多个文件构成一条 Session 记录
  - 必须填写「描述」「感受」「创建时间」三个必填字段才能归档
  - 可先暂存（待处理），后补充信息再归档
  - 全为文本文件时，描述字段由文件内容自动填充

#### FR-08 粘贴文字
- **描述**：用户可直接粘贴纯文本内容，系统自动保存为 `.txt` 文件
- **规则**：文字内容自动填入描述字段，无需手动填写

#### FR-09 记录编辑与归档
- **描述**：待处理记录可随时补充/修改字段，信息完整后可归档
- **规则**：
  - 已归档记录的修改会生成 `edit_history` 条目，记录变更前后的字段值
  - 归档记录自动生成对应的 `.md` 文件

#### FR-10 评论区
- **描述**：每条记录（无论待处理/已归档）均可添加、删除评论
- **规则**：
  - 评论记录评论者 `user_id`，渲染时显示用户名
  - 评论支持随时追加和删除，无需记录处于特定状态

---

### 5.4 时间锁权限状态机

#### FR-11 可见性控制
- **描述**：每条 Session 拥有独立的 `visibility` 状态
- **三态**：
  - `private`：仅创建者可见（默认）
  - `pending_unlock`：已申请共享，等待满足时间条件
  - `shared`：对方可在「情侣空间」中查看

#### FR-12 申请共享（时间锁触发）
- **描述**：用户可对私密记录申请共享给对方
- **规则**：
  - 申请后记录变为 `pending_unlock`，记录申请时间
  - **解锁条件**：记录的 `upload_time` 距今满 90 天（而非申请时间）
  - 系统每次加载时自动检查并推进满足条件的记录至 `shared`
  - `pending_unlock` 阶段可随时撤回，记录恢复为 `private`
  - 撤回后再次申请，已流逝的时间仍然计入，不重置计时

#### FR-13 情侣空间（共享内容查看）
- **描述**：「情侣空间」Tab 展示伴侣共享给自己的记录
- **规则**：
  - 仅展示伴侣 `visibility == "shared"` 的记录
  - 视图层使用 `can_view_session()` 进行二次权限校验
  - 查看方只能阅读和评论，不能编辑

---

### 5.5 分手协议与数据冻结

#### FR-14 单方发起解绑
- **描述**：任一方可在「账户」页发起分手流程
- **规则**：
  - 触发后关系状态变为 `frozen`，记录冻结到期时间（90 天后）
  - 冻结期内应用全局只读，所有上传/编辑入口禁用
  - 双方权限维持冻结前状态不变
  - 冻结到期后系统自动销毁该情侣关系下的全部数据

#### FR-15 双方同意立即解绑
- **描述**：双方均同意时，立即销毁全部数据
- **规则**：无需等待冻结期，操作不可逆

#### FR-16 冻结期数据导出
- **描述**：冻结期内，用户可导出属于自己的文件
- **规则**：
  - 仅导出 `session.user_id == 当前用户` 的文件
  - 不包含对方的任何记录、双方共同生成的报告
  - 导出格式：ZIP 压缩包，点击下载

#### FR-17 数据销毁
- **描述**：关系解除时，销毁该情侣关系下的全部数据
- **范围**：所有 sessions、磁盘文件、生成的 `.md` 文件
- **规则**：操作不可逆；双方 `couple_id` 重置为 `null`

## 6. 功能需求（第二阶段）

> 第二阶段尚在规划中，以下为功能方向定义，不作为当前版本交付承诺。

### 6.1 文本 RAG 知识库

#### FR-18 向量化索引
- 以 Session 为基本 Chunk 单元建立向量索引
- 将 `description`、`feeling`、`content_time`、`visibility`、`user_id` 作为 Metadata
- **强约束**：只允许对 `visibility == "shared"` 的记录建立索引，私密记录严禁进入向量库

#### FR-19 检索过滤
- 所有 RAG 检索调用前强制过滤：`couple_id == 目标情侣` AND `visibility == "shared"`

---

### 6.2 智能体模块 A：定期情感周报

#### FR-20 周报生成规则
- 触发方式：定期（如每两周）或用户主动请求
- 数据来源：**严格限定**为 `visibility == "shared"` 的记录
- 输出格式：使用抽象的情感温度计指标，不引用具体事件原文
- 数据平衡：当双方数据量差异显著时，报告须客观指出，不偏袒数据较多的一方

#### FR-21 周报内容约束
- 禁止在报告中引用单方私密记录的内容
- 禁止做对错裁判，只做情感趋势的客观描述

---

### 6.3 智能体模块 B：实时情感助手

#### FR-22 对话入口
- 为每位用户提供独立的私密对话入口
- 用户可就情感困惑、关系矛盾向助手提问

#### FR-23 中立性约束（System Prompt 强制要求）
- 助手严禁对双方行为做对错判断
- 回复强制遵循 NVC（非暴力沟通）四步框架：**观察 → 感受 → 需要 → 请求**
- 只做情绪安抚和客观信息映射，不充当任何一方的辩护律师
- 即使用户强烈要求裁判，也只重申 NVC 原则，不妥协

#### FR-24 数据访问约束
- 助手只能调用 `visibility == "shared"` 的数据作为上下文
- 严禁在回复中直接引用对方记录的原文
- 单方提问时，助手不得向提问者透露对方私密记录的内容

## 7. 非功能性需求

### 7.1 隐私与安全

| 需求 | 当前实现 | 目标实现 |
|------|----------|----------|
| 密码存储 | SHA-256 + 固定盐 | bcrypt / argon2 |
| 数据传输 | 局域网明文（HTTP） | HTTPS（生产环境） |
| 数据隔离 | DB 层按 `user_id` 过滤 | 行级权限控制 |
| Token 安全 | URL query param，24h 过期 | HttpOnly Cookie |
| 文件访问 | 本地路径，无鉴权 | 签名 URL / 鉴权中间件 |

### 7.2 性能

- 页面首次加载：< 3 秒（本地环境）
- 文件上传（100MB 以内）：不阻塞 UI，有进度反馈
- DB 读写：单文件 JSON，适合 < 1000 条记录的 Demo 场景

### 7.3 可用性

- 支持 PC 浏览器（Chrome / Edge）访问
- 局域网内任意设备可通过 IP:Port 访问
- 无网络连接时应用仍可正常运行（纯本地）

### 7.4 可维护性

- 三层模块完全解耦，替换任意层不影响其他层
- `FIELD_SCHEMA` 驱动 UI 渲染，新增字段无需修改其他代码
- `backend/db_manager.py` 可直接复用为后端 API 的数据访问层

## 8. 数据模型摘要

### 核心实体与关系

```
User (1) ──────────── (N) Session
  │                          │
  │ belong to                │ belong to
  ▼                          ▼
Couple (1) ─────────── (N) Session
```

### 实体关键字段

**User**：`user_id`、`username`、`password_hash`、`couple_id`、`joined_at`

**Couple**：`couple_id`、`user_a`、`user_b`、`couple_status`（pending_bind / active / frozen / dissolved）、`freeze_ends_at`

**Session**：`session_id`、`user_id`、`couple_id`、`status`（pending / final）、`visibility`（private / pending_unlock / shared）、`upload_time`、`unlock_requested_at`、`shared_at`、`files[]`、`comments[]`、`edit_history[]`

**AuthToken**：`token`、`user_id`、`expires_at`

### 关键状态机

```
【可见性】  private → pending_unlock → shared
【情侣关系】 pending_bind → active → frozen → dissolved
【记录状态】 pending → final
```

## 9. 边界与限制（Out of Scope）

以下内容**明确不在当前版本范围内**：

| 排除项 | 原因 |
|--------|------|
| 多对多关系（多人群组） | 产品定位为严格双人，多人会破坏隐私边界 |
| 公开/分享给第三方 | 违背"情侣专属私密空间"的核心定位 |
| 云端存储与同步 | Demo 阶段，本地优先；云端方案留待 Beta 阶段设计 |
| 消息推送与通知 | 当前为被动刷新模式，主动推送需独立服务支持 |
| AI 生成内容的编辑/删除 | 第二阶段 Agent 输出为只读报告 |
| 用户名/密码修改 | Demo 阶段简化，生产版本需补充 |
| 账号注销 | Demo 阶段暂不支持，需通过分手协议数据销毁间接实现 |

## 10. 成功指标

### Alpha 阶段（当前）

| 指标 | 标准 |
|------|------|
| 核心功能可用性 | 注册 → 绑定 → 记录 → 申请共享 → 查看全流程可跑通 |
| 隐私隔离验证 | 用户 A 的私密记录对用户 B 不可见（测试验证） |
| 分手协议验证 | 冻结/销毁流程不留残余数据 |
| 登录持久化 | 刷新页面后不退出登录 |

### Beta 阶段（目标）

| 指标 | 标准 |
|------|------|
| 实际使用留存 | 两名真实用户连续使用 30 天 |
| 内容积累量 | 每对情侣平均 20+ 条 Final 记录 |
| 时间锁触发 | 至少 1 条记录完成 90 天到期自动解锁验证 |

### Phase 2 阶段（目标）

| 指标 | 标准 |
|------|------|
| 周报质量 | 用户认为报告内容客观，未泄露对方隐私 |
| NVC 约束 | 助手在情感冲突问题上不做裁判（人工评估） |
| RAG 相关性 | 检索结果前 3 条中至少 2 条与查询语义相关 |

## 11. 风险与约束

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| JSON 单文件并发写入冲突 | 数据损坏（低概率，Demo 场景） | Beta 阶段迁移至 SQLite 或关系数据库 |
| SHA-256 固定盐密码方案 | 彩虹表攻击（本地环境风险极低） | 生产环境替换为 bcrypt/argon2 |
| Token 存于 URL 可被浏览器历史记录 | 同设备共用时存在隐患 | 生产环境改为 HttpOnly Cookie |
| `db.tick()` 依赖页面加载触发 | 冻结期到期时若无人访问则延迟销毁 | Beta 阶段加入独立定时任务 |
| 本地路径硬编码 | 换机器部署需手动修改路径 | 使用 `Path(__file__).parent` 动态解析，已规避 |
| LLM 中立性 | 第二阶段 AI 助手可能被诱导偏袒 | System Prompt 严格约束 + 定期人工抽查 |
