# OurPresent 上线路线图

**最后更新**：2026-05-16
**性质**：从当前 Alpha 走到商业化上市的阶段性计划。**关键作用是把"卡在用户决策 / 资源 / 法律手续"的事情显式列出来**，避免架构师默认等待、用户默认我在动。
**前置文档**：`docs/DIRECTION.md`（产品方向）·`docs/E2EE.md`（协议设计）·`docs/PHASE2.md`（Phase 2 执行细节）

## 阶段概览

| 阶段 | 状态 | 卡点（谁来解锁） |
|------|------|----------------|
| Phase 0 · Alpha 本地 Demo | ✓ 完成 | — |
| Phase 1 · Alpha UX 收尾 | ✓ 完成（v3.1.0） | — |

## Phase 0 · Alpha 本地 Demo（当前状态）

**做完的事**：

- Streamlit + SQLite 单进程，整库 dict 模型，71+ 测试
- 核心机制（延时共享 / 冻结期销毁 / 三视图）跑通
- 合成数据流水 + 角色卡 + 剧本（产物保留作未来回放基线）
- E2EE 协议设计已落定（`docs/E2EE.md`）
- 小程序事实边界已调研（`docs/research/wechat-miniprogram-crypto.md`）

**这阶段不会再增添：** AI 模块、PG 迁移、合成流水线扩展。所有这些都已冻结，等 Phase 2 重新决定形态。

## Phase 1 · Alpha UX 收尾（我可独立驱动，无外部依赖）

**目标**：在没有任何云端 / 小程序投入的情况下，把延时表达机制本身的 UX 打磨到"找两个真朋友坐下来用 10 分钟就能 get 到"的程度。

**为什么先做这个**：
- 延时表达是产品灵魂卖点，Phase 2 之后再回头改 UX 成本翻倍
- Streamlit 是当前最快的 UX 迭代载体；mini-program 工程一旦启动，每改一个交互都要走前后端
- 这阶段产出的 UX 决策直接转译成 mini-program 的设计稿，没有浪费

**候选工作**（我可以列详细任务卡）：
- 写入流程：感受 / 原因 / 解锁时间这几个字段的填写顺序与默认值
- 时间锁的可视化（90 天倒计时、解锁前 1 天提醒、即将解锁的 affordance）
- 解锁瞬间的 UX（已读回执？情绪标记？评论入口的可见度）
- 冻结期销毁的"提示 → 反悔 → 销毁"完整链路
- 三视图（我 / 我们 / 待解锁）的切换成本与信息密度
- 新用户引导（首次配对 → 第一条记录 → 第一次解锁的"魔法时刻"）

**完成判据**：你自己拉一个朋友坐下来不用解释就能上手。

## Phase 2 · 0→1 本地最小闭环（端到端 E2EE）

**战略转向 2026-05-16**：不再走"境外快速开发 / 后期迁回境内"路线。境内路线锁定，**本地优先**：M0~M6 全部在本地 Docker Compose + 内网穿透真机联调完成，上云只是配置切换（DB / 对象存储 / API endpoint 三个环境变量 + HTTPS 证书）。

**目标**：把 E2EE 协议从纸面跑到本地两台真机，证明小程序 + 后端 + 密钥协商 + 密文存储 + 延迟派送的端到端链路可工作——服务器、备案、域名都不阻塞这一步。

**详细执行细节见 [`docs/PHASE2.md`](./PHASE2.md)**，本节只列骨架。

### 用户必须提供的事情（极简）

| 项目 | 状态 | 说明 |
|------|------|------|
| 微信小程序 AppID | 用户去申请 | 接口测试号即可（免主体免备案） |

服务器 / 域名 / ICP **都不在 Phase 2 路径上**。学长的服务器不推荐共用（隐私污染面 / 故障域耦合 / 备案归属），等 Phase 2 跑通后再单独要一台。

### 我会驱动的事情

1. `server/` + `miniprogram/` 进同 mono-repo，与 `backend/` `frontend/` 共存
2. libsodium.js 在真机 + 开发者工具的 smoke test（M1）
3. FastAPI + PG + Minio + Nginx 本地 Docker Compose 骨架（M2）
4. 配对协议 / SAS 短码 / WebSocket 推送（M3）
5. 客户端加密写入 + 服务端延时派送门禁（M4）
6. 伴侣协助恢复 + 用户口令备份恢复（M5）
7. 冻结期销毁 + 本地内测自验收（M6）

### 完成判据

架构师 + 用户两台真机通过 cloudflared 临时域名跑通自验收清单（详见 `docs/PHASE2.md` M6）：配对 → 延时写入 → 解锁 → 评论 → 冻结 → 销毁全跑通；PG / Minio 内容字段抽检全部不可读；伴侣协助 + 口令两路恢复都成功。

### Phase 2 不做的事

- 上云（Phase 2.5 才做）
- AI 模块（仍冻结）
- 复杂 UX（按最简能跑通即可，UX 在 Phase 3 打磨）
- 多设备同步（一人一台手机就够）
- 性能优化 / 监控 / 报警（够内测就行）

## Phase 2.5 · 上云切换（备案下来后）

**预估工作量 ≤ 1 天**，纯配置切换。详见 `docs/PHASE2.md` 「上云切换」节。

### 用户必须解锁

- ICP 备案下来（用户在办）
- 独立云服务器（**不与学长项目共用**，单独要一台）
- 域名（备案附带或单买）

### 我会驱动的事情

- 同 `docker-compose.yml` 在云机起，三个环境变量切换（`DATABASE_URL` / `OBJECT_STORE_ENDPOINT` / `API_BASE_URL`）
- Minio → 阿里 OSS / 腾讯 COS（S3 协议层不变）
- Let's Encrypt 自动证书 + Nginx
- 小程序"request 合法域名"换正式域名
- 真机回归 M6 自验收清单

## Phase 3 · 私测（体验版 + 真用户）

**目标**：把 Phase 2 + 2.5 跑通的机制放给 5-10 对真情侣用 4-8 周，收集反馈。

### 用户必须解锁

- **加入测试名单的情侣**：微信小程序体验版限制最多 100 个测试者；你需要拉到 5-10 对愿意把私密记录交给一个内测产品的情侣（这一步可能比技术还难）
- **客服 / 反馈渠道**：内测期间一定会有数据丢失 / 配对失败 / 解锁异常等支持需求，你需要决定谁回复（你自己 / 我作为副本 / 还是自动化 FAQ）

### 我会驱动的事情

- UX 打磨（基于 Phase 1 决策 + Phase 2 真机反馈）
- 错误处理 / 文案 / 引导
- 监控基础（错误率、配对成功率、解锁成功率、数据丢失上报）
- 隐私协议 / 用户协议**初稿**（法务审在 Phase 4）

**完成判据**：5-10 对情侣里有 ≥ 60% 在 8 周后仍在用，且没有发生过"内容明文泄露"事件。

### Phase 3 不做的事

- 收费（仍免费）
- 公开宣传（仍口口相传）
- AI 模块

## Phase 4 · 公测 / 上市

**目标**：转商业化主体，向公众开放。

### 用户必须解锁

| 项目 | 说明 |
|------|------|
| **法人主体** | 公司或个体工商户。**这是无法绕开的**：小程序公开发布、商业化收款、ICP 备案到公司名下都需要 |
| ICP 备案（公司名下） | 如果 Phase 3 已经办了个人备案，到这一步要变更主体 |
| 微信小程序认证（公司主体） | 几百元/年，发布必需 |
| 公司银行账户 + 微信支付商户号 | 未来 AI 模块收费的基础 |
| 隐私协议 / 用户协议法务审 | 这是 Tier 1 隐私承诺真正变成法律护盾的关键步骤——文案里写的"我们也看不到"在法律上要站得住 |
| 客服 / 运营资源 | 公测后日活上 X 之后无法只靠你一个人回复 |

### 我会驱动的事情

- 性能 / 可扩展 / 监控加固
- 多设备同步实装（一个人多台设备共享 `K_couple`）
- 安全审计准备材料（如有第三方审）
- AI 模块**正式启动设计**（产品方向里推后的部分，到这阶段重新评估）

## 阶段间的"切换" checkpoint

- **0 → 1**：✓ 已完成
- **1 → 2**：✓ 已完成（v3.1.0 发布，Phase 2 启动）
- **2 → 2.5**：本地 M6 自验收清单全过 + 备案下来 + 用户解锁独立服务器
- **2.5 → 3**：上云后真机回归通过 → 开始招测试者
- **3 → 4**：内测留存率达标 + 法人主体 + 法务初审过

## 我不知道但需要你告知的事情

1. **法人主体当前状态**：你已经有公司 / 个体户了吗？还是完全个人？这决定 Phase 4 的准备周期（公司注册要 4-6 周）
2. **上市目标时间感**：你心里的"上市"是 6 个月内 / 1 年 / 没定？这决定我现在是按"省成本"还是"省时间"模式来推进

这两件事不告诉我也不阻塞 Phase 2，但越早告诉我，越能让中长期计划准确。
