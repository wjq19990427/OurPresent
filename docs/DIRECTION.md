# 产品方向与隐私要求（架构师与实现工必读）

**最后更新**：2026-05-13
**性质**：高层战略前提，比任何任务卡都优先级高。任何技术决策与本文档抵触时，**先停下来与用户对齐**，再决定改方向还是改本文档。

## 商业化路径

- 本项目**面向终端消费者**（情侣），不是开源 / 开发者工具
- 分发形态目标：**微信小程序**（以及可能的原生 App），不是 GitHub clone
- 数据同步**必走云端**，没有"本地部署"这种选项；所有"本地 SQLite" / "本地 Demo" 是 Alpha 临时形态，不是终态

## 卖点排序

1. **延时表达机制本身** —— 这是市场需求点、是用户买单的理由
2. **"我们也看不到"的隐私承诺** —— 第二条卖点；情侣亲密记录赛道里这句话**本身就是产品资产**，不是技术细节，是用户做付费决策时的安全感来源
3. **AI 辅助（周报 / NVC / 词典等）** —— **暂缓**。先把延时表达 + 隐私做到上市，AI 等产品成熟后再向用户开放

## 付费节奏

- **延时共享 + 隐私承诺：免费**。这两条是品牌资产，不设付费门槛，所有用户默认享有
- **AI 模块：未来付费**。引入 AI 时再设计付费层，并解决 AI × E2EE 的明文访问问题

## 产品原则

- **"我们也看不到"不能因为工程难度而打折扣**。架构 / UX / 文案都围绕这条承诺生长
- **AI 暂缓与隐私底线强一致**：服务端跑 AI 要读明文，与 E2EE 冲突。AI 不是被牺牲，是被排到正确的时间窗
- **任何 server-side 读取用户内容（搜索、统计、AI、通知）的需求，现阶段一律拒绝或推后**

## 隐私底线 —— 端到端加密分档

### 决策（2026-05-13 锁定）

**架构按 Tier 1 设计，预留 Tier 2 退路，Tier 3 仅作最后兜底**。

| 档位 | 钥匙位置 | 服务器是否技术上可解密 | 对外承诺 |
|------|---------|---------------------|---------|
| **Tier 1（默认目标）** | 仅用户设备 | 不能 | "我们也看不到" |
| **Tier 2（退路）** | 服务器存"被用户口令加密过"的钥匙密文 | 不能（除非拿到用户口令） | "我们也看不到，但口令丢了就找不回了" |
| **Tier 3（兜底）** | 服务器明文持有 | 能 | "加密存储、严格权限、承诺不窥探" |

Tier 1 与 Tier 2 在数据流上几乎一致（服务端只存密文 + 元数据），差别仅在"钥匙存哪"——架构上可延后决定，不阻塞前期实现。Tier 1 → Tier 3 是巨大重构，反向几乎不可能，因此宁可前期多投入也要走 Tier 1。

### 时间锁机制 × E2EE 的关系（已澄清）

**时间锁的可信性来自服务端控制密文派送时机，不来自服务端持有钥匙**。两件事不一样：

- 配对时双方设备建立共享密钥 `K_couple`，服务端**永远**不持有
- 撰写方加密好内容上传，密文标记 `unlock_at`
- 服务端持有密文，但**拒绝把这条密文派送给接收方设备**，直到 `unlock_at`
- 撰写方自己看自己的内容随时可以
- 到点之后服务端才把密文流向接收方的所有设备
- 元数据派发要克制：可以让接收方知道"周五会有内容到"，不能泄露主题、感受等任何内容字段

这就是真 E2EE。服务端是"延迟派送的快递柜"，全程拿不到明文。

### 设备丢失 / 账号恢复策略（已敲定）

**首选：伴侣协助恢复**

1. A 丢手机，在新设备上登录、触发恢复请求
2. 服务端通知 B 设备
3. **必须带外（OOB）确认**：A 通过物理见面 / 通电话向 B 念服务端发的短码，B 验证通过后才在设备上点确认。防冒充
4. B 的设备用 A 新设备的临时公钥加密 `K_couple`，**经服务端中转**给 A 的新设备
5. A 新设备解出 `K_couple`，旧设备公钥在服务端被吊销
6. 服务端全程只看到密文中转，钥匙不落地

**鼓励但不强制：用户自控备份**

- 把 `K_couple` 用用户自选口令派生密钥加密后导出到用户自己的微信云 / iCloud / 本地文件
- 服务端永远不碰
- UX 文案重点：**"这是我们对你隐私的承诺的代价。强烈建议你启用备份。"**——把这条做成产品特征，**表面是保险，实则是隐私绝对安全的宣传抓手**

**边界 case：A 和 B 同时丢设备 / 同时清掉小程序 = 数据丢失，接受这个代价**。在 UX 文案里坦诚说明，把它转化为"看，连我们都救不回——这就是真 E2EE 的代价，也是真 E2EE 的证据"。

## 威胁模型（产品方明确的优先级）

- **首要防御目标：法律安全**。出现重大事故 / 司法调取请求时，我们**技术上拿不出明文**，因此在法律上无能为力——这是 Tier 1 的核心法律价值，避免被用户起诉导致破产或刑责
- **次要：黑客拖库**。Tier 1 下数据库被拖只能拿到密文 + 元数据，无内容价值
- **当前不重点防范**：内部员工恶意（产品方对自己员工完全信任）。但 Tier 1 架构副作用是即便想看也看不到，自动达成
- **不在范围内**：国家级强制性后门要求。出现这种情况已是法律层面问题，超出技术承诺范围

## 立项前还需解决的硬问题

以下两条 Tier 1 实现绕不开，在开 E2EE 架构决策卡前需要有明确方案：

1. **情侣配对的密钥交换协议**：必须防中间人（OOB 短码 + 扫码典型形态）
2. **微信小程序的加密 / 密钥存储能力实测**：WebCrypto 支持范围、`wx.setStorage` 安全边界、长期闲置被驱逐的恢复路径——立项早期半天工作量做一次摸底

设备丢失策略与时间锁机制已锁定，不再讨论。
