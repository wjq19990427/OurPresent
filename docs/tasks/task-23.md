# Task 23: 微信小程序作为 E2EE 客户端的可行性调研

## 变更说明

类型：调研

输出一份事实报告，回答"微信小程序作为 OurPresent 端到端加密客户端，到底能做什么 / 不能做什么 / 被驱逐时会发生什么"。后续配对协议与设备恢复 UX 的设计都依赖这份报告的事实边界。

---

## 背景

`docs/DIRECTION.md` 已锁定：架构按 Tier 1 E2EE 设计（钥匙仅在用户设备，服务端拿不到明文），分发首选微信小程序。

**配对协议与设备丢失 UX 的设计空间，被小程序的真实能力边界倒过来约束**：

- 如果 `wx.setStorage` 在长期闲置被驱逐场景里频繁掉钥匙 → 配对必须做得很轻，"自助重新配对"要常态化
- 如果小程序里没有 secure-enclave 类设施 → 钥匙保护级别天花板就在那
- 如果加密库打不进主包 / 分包 → 加密算法选型受限
- 如果扫码 + 短码这类带外通道可用 → 配对的中间人防御就有抓手

本卡不做任何协议设计，只**用证据回答事实问题**。

## 要回答的问题

以下 8 个问题每条都必须有"结论 + 证据来源"两部分：

1. **加密 API 可用范围**：小程序里的 `wx.getRandomValues` / 自带 crypto / 第三方 JS 加密库各能做什么？AES-GCM / ECDH (P-256 或 X25519) / HKDF / Argon2 类 KDF 哪些可用、哪些必须自带库实现
2. **`wx.setStorage` 语义**：单条容量、总容量、是否系统层加密、是否跨小程序隔离、root / 越狱设备下能否读出明文、不同微信版本 / iOS vs Android 差异
3. **更安全的本地钥匙位置**：小程序生态里有没有类似 iOS Keychain / Android Keystore 的安全持久化接口？云开发 (`wx.cloud`) 能否做"用户口令派生密钥本地解开"的容器？还是只能落到 `wx.setStorage`
4. **钥匙丢失触发条件**：列举所有可能让本地存储被清空 / 不可访问的场景——用户主动清缓存、微信主动清理长期未用小程序、卸载重装、跨设备登录、隐私清理、系统更新。每条标注是否常见
5. **打包体积约束**：主包 2MB / 分包上限是多少？常见加密库（libsodium-js / forge / Noble Curves / WebCrypto polyfill）体积大约多少？落进体积后还剩多少给业务
6. **加密性能基准**：典型中端 Android + 主流 iPhone 上，1KB 文本 AES-GCM 加解密、一次 ECDH 协商、一次 Argon2 KDF 大约多少 ms。如能在开发者工具里跑 benchmark 最理想；做不到就引用社区实测
7. **用户自控备份载体**：把 `K_couple` 用用户口令加密后导出，可选位置 —— 微信云 (`wx.cloud`)、用户自己的网盘 / 文件、复制到剪贴板让用户自存。每种的隐私语义（服务端能不能看到密文 / 能不能看到口令 / 用户操作成本）
8. **带外（OOB）配对通道**：小程序的扫码 / NFC / 蓝牙 / 屏幕显示短码等能力。配对场景下"两人物理在场"能做到什么、做不到什么
