# Task 24: Alpha UX 收尾 · 核心写-锁-解锁旅程

## 变更说明

类型：优化（用户可见）

把延时表达机制的核心旅程——"写记录 → 等解锁 → 解锁瞬间"——的 UX 打磨到第一次见到产品的情侣也能感受到"时间在流逝"。当前 Streamlit Alpha 在这条主线上有几个体验缺口：默认时间锁选项不直观、待解锁记录在伴侣端完全不可感知（直到解锁瞬间也没有提示）、评论区在 visibility 切换之间的可见度规则不一致、写新记录入口的发现性较弱。本任务在不改 SQLite schema、不引入新数据字段的前提下，**仅通过 UI 文案 / 排版 / 状态展示**优化这条旅程。

---

## 背景

`docs/ROADMAP.md` Phase 1 候选第 1~3 项与第 7 项落地。本卡是 Phase 1 三张串行卡（W1/W2/W3）中的第一张，**核心约束**：

- **不改数据库 schema、不改 application 层公开签名**：所有改动都是 UI 层的展示与文案
- Phase 1 的产出会直接转译成 mini-program 的设计稿，所以**判断"好不好"的标准不是 Streamlit 漂亮不漂亮，而是同样的字段顺序、文案、状态名拿到 mini-program 也合理**
- 完成判据：你拉一个没用过本产品的朋友坐下来 10 分钟，他能不问就理解"为什么有延时"、"在等什么"、"解锁那一刻发生了什么"

涉及现状摘录（实现工可读代码确认）：

- 写新记录入口：`tab_mine` 顶部 `expander("✍️ 写新记录")`，默认折叠
- 申请共享默认值：当前默认"1 周后"（见 `components._unlock_at_for_choice` 调用处）
- 待解锁状态徽章：`[倒计时·还有 N 天]`（见 `_status_badge`）
- 详情区可见性徽章：`🔒 私密` / `⏳ 待解锁（还需 N 天）` / `✅ 已共享`（见 `_visibility_badge`）
- 评论区：`render_comments` 在所有 visibility 下都展示且都可写

## 要做的改动

每条都要落到具体 UX 改动（实现工自己定细节，但目标和约束在这里）：

1. **写新记录入口的发现性**：当前在 `tab_mine` 顶部以折叠 expander 形态出现，对新用户不够主动。改造目标是让"首次进入「我的」tab 且当前用户没有任何记录"时能更明显地引导写第一条。已有记录后再回到当前折叠态可接受
2. **写新记录字段顺序**：审视当前 `FIELD_SCHEMA` 在 `render_field_inputs` 下的渲染顺序，结合产品旅程（"什么打动了你 → 当时的心情 → 想说给对方的话 → 想让对方什么时候看到"），调整为对首次填写者更顺手的顺序。如需要可在 `render_field_inputs` 的 `prefix` 控制下做"写新记录用一套顺序、编辑用原顺序"的差异化
3. **申请共享默认值**：当前默认"1 周后"过短。结合产品方向（延时是产品魂，更长的等待更有戏剧性），选一个更能体现产品理念的默认值并在卡内说明理由（比如"30 天后"或"90 天后"）。约束：UI 上仍保留"立即 / 1 天 / 3 天 / 1 周 / 1 月 / 90 天 / 自选"全部选项
4. **待解锁状态徽章文案**：`[倒计时·还有 N 天]` 偏工程感，改为更有共鸣的表达。**禁止使用 emoji 堆砌**或"还有 N 天就能看到啦~"这种二次元口吻；目标是温柔克制
5. **解锁前预告（伴侣端）**：在「我们」tab，**伴侣的待解锁记录虽然内容不可见，但「即将解锁」这件事应让伴侣感知到**。设计一个轻量、不暴露内容的预告区块（如"明天会有一份给你"、"还有 3 小时"），放在 `render_us_tab` 共享时间线上方或某个合理位置。**绝对不能泄露作者写了什么 / 上传了哪类文件 / 任何描述性字段**——仅"有一份待解锁"+"解锁时间"两个信号
6. **解锁瞬间的标识**：在「我们」tab，**解锁后 24 小时内的记录**在卡片上有清晰可辨的"新"视觉区分。`shared_at` 字段足以判断（"now - shared_at < 24h"）。无需新增数据字段
7. **评论区位置 + 可见度规则**：当前「我们」tab 的实现里，**所有 session 共用 timeline 末尾的全局详情区**——点击某条 session 卡 → 详情和评论被渲染在所有卡片**下方**的固定位置，用户必须滚动到底部才能看到评论。**这是本卡要修掉的核心 UX 缺陷**。改造目标：评论区跟随其所属 session 就近展示（卡片下内嵌展开 / 卡片内 expander / 紧邻该卡的容器，实现工选具体形态，但**禁止再回到 timeline 末尾的全局位置**）。**同时**重审 visibility 下的可见度：建议**私密记录**评论区只对作者本人可见可写；**待解锁记录**评论区完全隐藏（未解锁前评论无意义）；**已共享记录**评论区双方可见可写。在卡内说明你的选择
8. **「我们」tab 阅读态减负**：当前 `render_card` 对每张卡都展示 `✅ 信息完整` 或 `⚠ 待补充：xxx`——这是**写作者**视角的元信息，**不应出现在共享时间线**（阅读者只想看内容），严重影响观感。在「我们」tab 渲染卡片时隐藏完整度提示；「我的」tab 保留以辅助作者补完
9. **手机端作者归属仍可一眼区分**：之前 codex 已做过一轮 mobile-first 排版，但实测手机宽度下「我们」tab 两人的卡片都占满整屏、左右分栏视觉消失 → 无法一眼分辨"这条是我的"还是"伴侣的"。改造目标：在手机宽度下仍能瞬间识别作者归属。**禁止引入自定义 CSS / 第三方组件**，仅用 Streamlit 自带能力（如保留缩窄的分栏 / 头像 emoji / 用户名前置标签 / 颜色化 caption，由实现工选最克制的方案）。**桌面端不要因此回退**——既有的左右分栏视觉在宽屏下保留
10. **更新 `docs/api/frontend_streamlit.md`** + **`CHANGELOG.md` `[Unreleased]`** 1-2 句总结

## 验收（用户可见行为）

- 没用过本产品的朋友打开 Alpha → 进入「我的」 → 能自己找到写第一条记录的入口
- 写记录时字段填写顺序自然，到"想让对方什么时候看到"那一步时默认选项不再是"1 周后"
- 写完后立刻切到「我们」tab（用伴侣账号） → 能感知到"有一份待解锁"，但完全看不到内容线索
- 解锁时间到了 → 伴侣回到「我们」tab → 新解锁的记录在视觉上能一眼认出来
- 待解锁阶段的记录详情中不出现评论入口
- 已共享记录的评论区**紧邻该 session 卡**展示，不再统一堆在 timeline 末尾
- 「我们」tab 卡片不再出现"信息完整 / 待补充"提示
- 手机宽度下浏览「我们」tab，仍能一眼看出每条是自己还是伴侣写的
- 字段顺序、徽章文案、预告区块、评论位置、作者归属信号在 `docs/api/frontend_streamlit.md` 有对应记录

## 不要做

- **不要改数据库 schema / domain models / application 层公开签名**——纯 UI 层任务
- **不要新增"已读回执"或"情绪标记"功能**——这是后续卡的范围，且涉及 schema
- **不要新增独立的"待解锁" tab**——三视图重组是 W3 卡，本卡不动信息架构
- **不要触碰 `docs/STATUS.md`、`docs/DIRECTION.md`、`docs/ROADMAP.md`、`docs/E2EE.md`、`docs/PRD.md`、`CLAUDE.md`**——架构师独占
- **不要触碰冻结期销毁链路**——那是 W2 卡的范围
- **不要做新手引导 / onboarding 流程**——W3 卡的范围
- **不要为了"看起来更漂亮"加 emoji 装饰**——文案克制，符合"两个相爱的人之间私密记录"的基调
- **不要新增第三方依赖 / 自定义 CSS / Streamlit components**——保持 Alpha 简单
