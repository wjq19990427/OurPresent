## 接口约定

### 「我们」周报区状态矩阵

| 服务状态 | 最新报告状态 | 展示 |
|----------|-------------|------|
| 未绑定伴侣 | — | 周报区不显示 |
| 双方都未开启 | — | 邀请文案「一起开启情感周报，每周看到我们的足迹」+ 跳「设置」按钮 |
| 仅自己开启 | — | 「⌛ 等待对方一同开启」+ 提示对方在「设置」里的位置 |
| 仅对方开启 | — | 「对方已开启周报，要不要一起？」+ 跳「设置」按钮 |
| 双方都开启 | 无任何报告 | 「邀请你写下第一周的共享记录」（不显示「立即生成」字样，避免与 task-9 临时按钮混淆） |
| 双方都开启 | 最新 ready | 完整四模块渲染（已由 task-9 实现，本任务可调文案） |
| 双方都开启 | 最新 sparse | 仅 footprint + 一句温和陈述「这周共享记录较少，留些空白也好」 |
| 双方都开启 | 最新 failed | 显示「上一次生成遇到了一些波折，会在下次自动重试」；不显示报告内容 |
| 冻结期 | 任意 | 历史报告可读、不会生成新报告；展示一行说明 |

文案具体措辞由实现工敲定，保持温和、不评判、不指责的语气基调。

### 历史报告入口

- 位置由实现工自决（推荐放「设置」的周报 section 里，与历史列表语义匹配，也减少「我们」tab 的视觉负担）
- 展示形态：列表项展示 `window_start ~ window_end` + 状态徽章；点击展开完整四模块
- 排序：按 `generated_at` 倒序
- 不展示 `status == failed` 的历史项

### 设置 section 文案打磨

- 服务开关旁加一句话说明：「周报基于你们已共享的记录生成，不读私密内容」
- 频率 dropdown 选项文案：`7 天 / 每周`、`14 天 / 每两周`、`30 天 / 每月`（具体措辞实现工敲定）
- 解绑后状态：`weekly_report_enabled` 保留；下次绑定新伴侣后开关仍是上次值，频率回到默认 7 天（这是 task-7 的字段语义，本任务只是 UI 上让这一点不让用户困惑）

## 验收行为（用户视角）

`uv run streamlit run main.py` 跑通状态矩阵全部 9 行：

- 双方都关 → 「我们」给出邀请文案
- 仅一方开 → 两边各看到对应文案
- 双方都开但无报告 → 给出「写下第一周的共享记录」
- 制造 ≥ 3 条 shared，跑出 ready 报告 → 完整四模块
- 制造 < 3 条 shared，跑出 sparse 报告 → 仅 footprint + 温和提示
- 故意让 LLM 失败，跑出 failed 报告 → 显示等待重试文案，不暴露错误细节
- 进入冻结期 → 历史可看，不生成新报告
- 历史入口 → 倒序展示历史报告，点击展开完整内容
- 任务卡完成后**临时手动按钮仍在**（task-9 行为，待架构师另开任务删除）

自动化检查：

- `uv run python -c "import frontend; print('OK')"`
- `uv run pytest` 全绿
- `uv run ruff check .` 无错
- 全 UI 文案审视一遍：无「你应该」「你需要」「建议你」之类的祈使句；无对任一方的评判语

## 已知陷阱

- 9 行状态矩阵的边界容易遗漏，开发时建议**先把所有分支的占位文案写出来**，再回头打磨文案——比一边写一边想措辞效率高
- 「邀请对方开启」类文案要避免「对方还没开启」这种被动语态偏指责；保持「邀请」「一起」的主动协作语气
- 历史列表如果数据量大（实际上 alpha 阶段也大不到哪去），不做分页；后期 beta 再考虑
- 频率 dropdown 改动立即生效会影响 next tick；如果用户频繁改动可能导致 cron 判定混乱——但用户主动改动应当被尊重，不做防抖
