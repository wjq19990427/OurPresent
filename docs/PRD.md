## 1. 产品概述

### 1.1 产品定义

**OurPresent** 是一款专为情侣设计的私密双人情感记录与智能辅助工具。其核心价值主张是：

> 在保护双方绝对个人隐私的前提下，通过"延时共享"机制让感情沉淀，并在合适的时机以温柔的方式让彼此看见对方的心意。

产品分两个阶段构建：
- **第一阶段**：双人私密记录空间，配备时间锁权限引擎与完整的关系生命周期管理。
- **第二阶段**：接入 LLM + RAG，引入 AI 智能体作为关系的客观记录者和情感疏导者。

### 1.2 产品定位

| 维度 | 定位 |
|------|------|
| 核心功能 | 情侣双人私密内容记录、延时共享、情感 AI 辅助 |
| 差异化 | 时间锁机制（90 天延时）+ 严格隐私隔离 + NVC 中立 AI |
| 技术路线 | 本地化部署（隐私优先）→ 后期可选云端迁移 |
| 当前形态 | 本地内网 Streamlit Demo |

### 1.3 产品命名含义

"Present" 在产品中承载三层递进含义：

1. **我们的当下（Our Present）**：记录此刻的真实状态，无论喜悦还是难以直言的委屈。
2. **共同的馈赠（The Gift of Today）**：两人相遇本是礼物，提醒珍惜当下的陪伴。
3. **温柔地表达（Present to You）**：每一条记录经时间沉淀后，化作一份延迟解锁的情感礼物。

## 2. 目标用户与使用场景

### 2.1 核心用户画像

**用户 A：记录型一方**
- 感情细腻，有记录日常的习惯
- 有些心里话当面难以开口，倾向于用文字整理情绪
- 在意隐私，不希望自己的情绪被第三方（闺蜜/兄弟）外传
- 希望对方有一天能看到自己的感受，但需要时间缓冲

**用户 B：被动接收型一方**
- 可能不擅长表达，但愿意了解对方的内心
- 希望在双方关系稳定、情绪冷静后再了解对方的想法
- 对隐私有一定要求，不希望被强制分享

### 2.2 典型使用场景

| 场景 | 描述 |
|------|------|
| 日常记录 | 上传两人的照片、视频，配上感受描述，作为共同回忆存档 |
| 情绪倾诉 | 发生争吵后，一方记录下自己的委屈和想法，暂时私密存放 |
| 延时告白 | 记录一段深情的文字，申请 90 天后让对方看到 |
| 关系复盘 | AI 智能体基于双方共享内容，生成客观的情感周报 |
| 情感疏导 | 向实时助手倾诉情感困惑，获得 NVC 框架下的中立回应 |
| 分手处理 | 关系结束时，平静有序地完成数据清理，保护双方隐私 |

## 3. 核心问题与解决思路

### 3.1 现有痛点

1. **隐私外泄风险**：情侣间的矛盾一旦倾诉给闺蜜/兄弟，存在信息外传和被放大的风险，甚至激化矛盾。
2. **表达时机错位**：情绪激动时说出的话往往伤人，但等情绪平静后又难以主动开口。
3. **缺乏客观视角**：双方各执一词，缺少一个中立、不偏袒任何一方的视角来梳理关系状态。
4. **记忆碎片化**：两人的日常照片、视频散落在各自手机，缺乏统一的、有情感温度的归档空间。

### 3.2 解决方案映射

| 痛点 | 解决方案 |
|------|----------|
| 隐私外泄 | 本地化部署 + 严格的双用户隔离，记录默认私密，不经授权对方不可见 |
| 表达时机错位 | 时间锁机制：申请共享后满 90 天才对对方可见，给情绪足够的缓冲期 |
| 缺乏客观视角 | AI 情感周报 Agent：只基于双方共享内容，用抽象指标客观呈现关系状态 |
| 记忆碎片化 | 双人记录空间：图片/视频/文字统一归档，强制填写情感描述，形成结构化记忆 |
## 4. 阶段规划

| 阶段 | 目标 | 关键交付 | 状态 |
|------|------|----------|------|
| **Alpha** | 本地可运行 Demo | 注册/登录、情侣绑定、记录上传、时间锁、分手协议 | ✅ 已完成 |
| **Beta** | 稳定内网版本 | 前后端分离、更强密码方案、并发安全、移动端适配 | 🔲 规划中 |
| **Phase 2** | AI 智能体接入 | 情感周报 Agent、实时情感助手（NVC）、RAG 向量检索 | 🔲 规划中 |
| **v1.0** | 正式发布 | 云端部署选项、完整测试覆盖、用户引导流程 | 🔲 规划中 |

## 5. 功能需求（第一阶段）

### 5.1 用户账户系统

#### FR-01 用户注册

- 用户可通过用户名和密码创建账户
- 注册成功后获得唯一用户 ID

#### FR-02 用户登录

- 用户可通过用户名和密码登录
- 登录后进入自己的私有空间

#### FR-03 持久化登录

- 刷新页面后应尽量保持登录状态

### 5.2 情侣绑定系统

#### FR-04 发送绑定请求

- 用户可通过对方用户 ID 发起绑定请求

#### FR-05 接受或拒绝绑定

- 接收方可在界面中接受或拒绝绑定请求

#### FR-06 关系状态流转

- 绑定关系至少包含待绑定、已绑定、冻结、解除四种业务状态

### 5.3 内容记录系统

#### FR-07 上传文件

- 支持图片、视频等媒体文件上传

#### FR-08 粘贴文字

- 支持纯文字记录

#### FR-09 记录编辑与归档

- 用户可补充和编辑记录字段
- 记录可从待处理转为正式归档

#### FR-10 评论区

- 已共享或允许查看的内容支持评论交流

### 5.4 时间锁权限机制

#### FR-11 可见性控制

- 记录默认仅自己可见
- 对方不可直接读取未共享内容

#### FR-12 申请共享

- 用户可主动申请让记录进入共享流程
- 系统应基于既定规则决定何时真正对伴侣可见

#### FR-13 情侣空间

- 用户可查看当前对自己可见的伴侣内容

### 5.5 关系结束与数据保护

#### FR-14 单方发起解绑

- 任一方可单方发起关系解除流程
- 发起后进入冻结状态

#### FR-15 双方同意立即解绑

- 若双方同意，可立即完成解除并清理数据

#### FR-16 冻结期数据导出

- 冻结期内双方可导出自己的内容

#### FR-17 数据销毁

- 在满足规则后，系统应能彻底清理双方关系数据

## 6. 功能需求（第二阶段）

### 6.1 文本 RAG 能力

#### FR-18 向量化索引

- 对允许进入 AI 范围的文本建立索引

#### FR-19 检索过滤

- 检索结果必须遵守隐私与权限边界

### 6.2 智能体模块 A：情感周报

#### FR-20 周报生成规则

- 周报只基于允许访问的共享内容生成

#### FR-21 周报内容约束

- 输出应客观、克制，不替任何一方下判断

### 6.3 智能体模块 B：实时情感助手

#### FR-22 对话入口

- 用户可与助手进行实时对话

#### FR-23 中立性约束

- 助手不得扮演裁判或激化矛盾

#### FR-24 数据访问约束

- 助手不得读取不属于当前权限范围的私密内容

## 7. 非功能性需求

### 7.1 隐私与安全

- 默认私密，未共享内容不得越权访问
- 数据销毁流程应尽量彻底
- AI 阶段仍需遵守最小权限原则

### 7.2 性能

- Alpha 阶段以单机双用户顺畅使用为目标
- 常规页面操作不应出现明显卡顿

### 7.3 可用性

- 首次使用流程应尽量简单清晰
- 核心路径应覆盖注册、绑定、记录、共享、查看、解绑

### 7.4 可维护性

- 产品规则应可持续扩展
- 功能边界应清晰，支持后续演进

## 8. 边界与限制

以下内容不在当前版本范围内：

- 多人关系或群组空间
- 对第三方公开分享
- 正式云端同步与多端一致性
- 消息推送与通知服务
- 完整账号体系能力，如密码找回、账号注销等

## 9. 成功指标

### Alpha 阶段（当前）

| 指标 | 标准 |
|------|------|
| 核心功能可用性 | 注册 → 绑定 → 记录 → 申请共享 → 查看全流程可跑通 |
| 隐私隔离验证 | 用户 A 的私密记录对用户 B 不可见（测试验证） |
| 分手协议验证 | 冻结/销毁流程不留残余数据 |
| 登录持久化 | 刷新页面后不退出登录 |

### Beta 阶段（目标）

| 指标 | 标准 |
|------|------|
| 实际使用留存 | 两名真实用户连续使用 30 天 |
| 内容积累量 | 每对情侣平均 20+ 条 Final 记录 |
| 时间锁触发 | 至少 1 条记录完成 90 天到期自动解锁验证 |

### Phase 2 阶段（目标）

| 指标 | 标准 |
|------|------|
| 周报质量 | 用户认为报告内容客观，未泄露对方隐私 |
| NVC 约束 | 助手在情感冲突问题上不做裁判（人工评估） |
| RAG 相关性 | 检索结果前 3 条中至少 2 条与查询语义相关 |

## 11. 风险与约束

| 风险 | 影响 | 缓解方向 |
|------|------|----------|
| 隐私边界设计不清 | 伤害用户信任 | 所有新功能都先经过权限审视 |
| 表达辅助被误解为评判 | 激化矛盾 | 强化中立约束与产品文案边界 |
| 本地 Demo 承载能力有限 | 难以支撑更复杂场景 | Beta 阶段逐步补强数据层与部署能力 |
| 关系结束场景敏感 | 体验与信任风险高 | 冻结、导出、销毁流程保持清晰可解释 |
